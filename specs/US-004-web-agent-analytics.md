# US-004: Web Agent Analytics — Metric Definitions

## Status
- Prototype: Built (GBWebWidget.html, components: AnalyticsPanel, KpiTile, CompactFunnel, ChatsTable, CHATS_TABLE data)
- Production: Not yet shipped
- Sprint target: Phase 2 (alongside US-001 Dashboard)

## Problem statement
Before engineering can build the Dashboard tab (US-001), the team needs an agreed definition for every metric it will display. Without explicit definitions, the same word (e.g., "session") means different things to the AI provider, the embed script, and the Gallabox backend — leading to number mismatches that erode operator trust. This spec locks the definitions.

## User stories

### Story 1: Define and track "widget opened"
**As a** product analyst, **I want** a "widget opened" event to be recorded each time the widget becomes visible, **so that** we have a reliable baseline denominator for all funnel metrics.

**Acceptance criteria:**
- [ ] One "widget opened" event is fired per visibility trigger, regardless of whether a visitor sends a message
- [ ] The event captures: `open_type` (user_clicked | auto_opened), `source_url` (page URL), `timestamp`, `session_id`, `agent_id`
- [ ] Multiple opens by the same visitor in one page view count as one event (debounced at widget close/reopen within the same tab)
- [ ] The KPI tile label is "Widget Opened"

### Story 2: Define and track "session started"
**As a** product analyst, **I want** a "session started" event to be recorded only when a visitor sends at least one message after opening the widget, **so that** the session count reflects genuine engagement, not passive widget views.

**Acceptance criteria:**
- [ ] A session is counted when the visitor sends their first message (not on widget open)
- [ ] One session per widget open; a visitor who closes and reopens the widget in the same page view gets one session
- [ ] The KPI tile label is "Session Started"
- [ ] The session start event captures: `session_id`, `agent_id`, `open_type`, `source_url`, `timestamp`, `device_type`, `country` (from IP)

### Story 3: Define and track "contact captured" (identified vs anonymous)
**As a** product analyst, **I want** to distinguish sessions where we captured a contact record from those where we did not, **so that** we can measure lead capture rate independently of session volume.

**Acceptance criteria:**
- [ ] "Contact captured" = visitor submitted the lead form (name + phone) OR successfully completed a WhatsApp handoff that matched to an existing contact
- [ ] "Anonymous chat" = session that ended without a contact match
- [ ] `identified` count + `anonymous` count = total `sessions` (no overlap, no gap)
- [ ] Of identified contacts, "new contact" = contact record did not exist before this session; "returning contact" = record already existed
- [ ] The KPI tile label is "Contacts Captured"; anonymous is shown separately as "Anonymous Chats"

### Story 4: Define and track "CTA click"
**As a** product analyst, **I want** a "CTA clicked" event recorded each time a visitor clicks any CTA button in the widget, **so that** we can measure call-to-action effectiveness independently of whether it led to a session.

**Acceptance criteria:**
- [ ] One event per button click; multiple clicks in a session are counted individually
- [ ] CTA clicks are counted at the agent level, not per session (a session may have zero or multiple CTA clicks)
- [ ] The event captures: `session_id` (if one exists), `cta_label`, `cta_url`, `timestamp`
- [ ] CTA clicks are independent of handoff — a visitor can click a CTA without doing a WA handoff

### Story 5: Define and track "WA handoff"
**As a** product analyst, **I want** a "WA handoff" event recorded each time a visitor taps the WhatsApp continuation button and is routed to WhatsApp, **so that** we can measure how many widget sessions convert to WhatsApp conversations.

**Acceptance criteria:**
- [ ] One event per handoff action; if the visitor taps the handoff button twice (e.g., after navigating back), both are counted
- [ ] The event captures: `session_id`, `contact_id` (if matched), `source_url`, `entry_point_style` (header | starter | button), `timestamp`
- [ ] "Handoff converted" = a handoff that resulted in an active inbox conversation within 30 minutes (tracked as a separate sub-metric)
- [ ] Funnel step label is "WA handoff"; conversion rate = WA handoffs ÷ contacts captured

### Story 6: Lead score per session
**As a** Gallabox inbox agent, **I want** each session to have an AI-generated lead score (0–10), **so that** I can prioritize follow-up conversations by lead quality without re-reading the full chat transcript.

**Acceptance criteria:**
- [ ] Lead score is a number from 0 to 10 (inclusive), rounded to one decimal place
- [ ] Score is generated by the AI provider at session resolution time, not during the session
- [ ] The AI agent must return a structured JSON field `lead_score` alongside its session summary; if the field is absent, the session is marked "No score"
- [ ] Score is shown in the session log table (US-001 Story 6) as a numeric value
- [ ] Score is NOT shown in the KPI tiles; it is a per-session attribute only

### Story 7: Intent type per session
**As a** Gallabox inbox agent, **I want** each session to have an AI-extracted intent label (e.g., "Commercial Property", "Residential Buying"), **so that** I can quickly understand what the visitor was looking for before reading the full context card.

**Acceptance criteria:**
- [ ] Intent type is a short label (max 40 characters) extracted by the AI agent at session resolution
- [ ] The AI agent must return a structured JSON field `intent_type`; if absent, the session shows "Unknown intent"
- [ ] Intent type is shown in the session summary context card in the inbox (via Chat Infusion, US-002)
- [ ] Intent type is available as a column in the session log export

### Story 8: Average session duration
**As an** agent operator, **I want** to see the average duration of widget sessions, **so that** I can evaluate whether visitors are engaging deeply or bouncing quickly.

**Acceptance criteria:**
- [ ] Session duration = time from first message sent to last message sent in the session (not widget open to close)
- [ ] If a session has only one message, duration = 0
- [ ] Displayed as "Xm Ys" format (e.g., "2m 14s"); sub-minute sessions display as "Xs"
- [ ] Shown in the session log table; average across all sessions shown as a summary stat (future: KPI tile)
- [ ] Sessions without a resolved end time are excluded from duration calculations

### Story 9: Traffic source, device type, and country per session
**As an** agent operator, **I want** to understand where my widget visitors are coming from (referrer, device, geography), **so that** I can align widget configuration with my top acquisition channels.

**Acceptance criteria:**
- [ ] Traffic source is derived from the `Referer` header or UTM parameters at widget load time
- [ ] Device type is classified as Desktop, Mobile, or Other from the User-Agent string
- [ ] Country is derived from IP geolocation at session start; unknown geolocation shows "Unknown"
- [ ] These three dimensions are shown as breakdown cards in the Dashboard (US-001 Stories 5 and 9)
- [ ] Source URL (exact page URL where widget was loaded) is stored per session and shown in the session log

## Key decisions & trade-offs
| Decision | Chosen approach | Alternative considered | Reason |
|---|---|---|---|
| Session definition | At least 1 message sent | Widget opened | "Session = open" inflates the metric with passive impressions; we want sessions to mean genuine engagement |
| Lead score source | AI provider returns structured JSON with `lead_score` | Gallabox calculates score from conversation data | The AI already has full conversation context at resolution; offloading to the provider avoids duplicate processing |
| Handoff conversion window | 30-minute window to count as "converted" | Real-time only | Visitors may switch apps and return to WhatsApp minutes later; a 30-min window captures realistic conversion without false positives |
| CTA clicks vs handoff | Tracked as separate metrics | Combined into a single "conversion" metric | CTA (external redirect) and WA handoff are structurally different actions with different business value; combining them obscures performance |
| Anonymous session attribution | Included in session count, shown separately | Excluded from session count | Anonymous sessions represent real engagement even without a contact match; excluding them would undercount total interest |

## Notes for engineering
- In production, metric data comes from two sources: (1) Gallabox session event pipeline (widget opened, session started, contact captured, CTA click, WA handoff) and (2) AI provider completion logs (lead score, intent type, session summary). The prototype mocks all data locally.
- Lead score and intent type require the AI agent's system prompt to be configured to return a structured JSON block at session end. This is a prompt engineering dependency — engineering must coordinate with the AI/prompt team.
- `session_id` is the join key between Gallabox session events and AI provider completion logs. It must be passed to the AI provider at session start and returned in the completion log.
- Session duration calculation excludes time when the widget is minimised or the tab is hidden (use Page Visibility API). The prototype does not implement this; production must.
- All metric definitions above are the canonical production definitions. The prototype's mocked numbers satisfy these definitions (see the `AN` data comment at line ~2552) and can be used as a test fixture for the dashboard UI.
